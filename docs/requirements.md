# Requirements

Technical requirements for building, consuming, and contributing to Aptus CSS.

## Supported browsers

Aptus targets current versions of Chromium-based browsers (Chrome, Edge) and Firefox. Changes must be checked at minimum in a Chromium-based browser, and in Firefox when possible. There is no official support commitment for legacy browsers; `src/ui.css` includes an `#old-browser-warning` block that consuming projects can use to warn users on outdated browsers.

## Build tools

Aptus is built with two external CLI tools. They are not npm packages and are not listed in any `package.json`; there is no Node.js dependency for building this library.

- **rpack** — asset packer, responsible for minifying each CSS module listed in [rpack.json](../rpack.json) under `minifies.css:pages`.
- **rimi** — build and local deploy tool, configured in [rimi.json](../rimi.json). It copies `src/VERSION` and `src/APP_NAME` into the build output, runs `rpack build`, and can install the build into a local CSS library path (`install.path`, currently `$HOME/lib/css/{name}/{version}`).

Both tools must be available on the machine's `PATH` to build or deploy Aptus.

## Build commands

```bash
rpack build     # minify src/*.css into dist/*.min.css
rimi build      # copy VERSION/APP_NAME, then run rpack build
rimi package    # build and create a distributable package
rimi deploy     # build and install into the local CSS lib path
rimi info       # show current project name, version, and paths
```

## Source and distribution files

Source modules live in `src/` and are edited directly:

```text
src/reset.css
src/grid.css
src/image.css
src/commons.css
src/ui.css
src/notification.css
src/autocomplete.css
src/gdpr.css
src/datalist.css
src/breadcrumb.css
src/eyebrow.css
src/VERSION
src/APP_NAME
```

Each source module has a matching minified file generated into `dist/`:

```text
dist/reset.min.css
dist/grid.min.css
dist/image.min.css
dist/commons.min.css
dist/ui.min.css
dist/notification.min.css
dist/autocomplete.min.css
dist/gdpr.min.css
dist/datalist.min.css
dist/breadcrumb.min.css
dist/eyebrow.min.css
dist/VERSION
dist/APP_NAME
```

There is no single bundled `aptus.min.css`; the distribution is modular, one minified file per module. Files under `dist/` are generated by the build and must never be edited by hand. The unminified `src/*.css` files are the ones to use during development and debugging; the `dist/*.min.css` files are the ones consuming projects should link in production.

## Consumer integration

Aptus is not fetched by consuming projects directly from this repository or from `dist/`. The hand-off happens in two steps:

1. **`rimi deploy`** (run in this repository) builds Aptus and installs it into a local, version-pinned library path: `$HOME/lib/css/aptus/{version}/`. This is a machine-local install location, not a path any consuming project serves over HTTP.
2. **The consuming project's own build** copies the modules it needs from `$HOME/lib/css/aptus/{version}/` into its own deployed `res/css/` directory (never into its `src/`), alongside its own `res/img/` and `res/js/`. This copy step is owned by the consuming project, not by Aptus.

The `res/{css,img,js}` sibling layout described under "Image resources" below is guaranteed by step 2, which is why Aptus CSS can reference its own images with paths relative to the CSS file instead of an absolute or configurable path.

## Version

`src/VERSION` is the single source of truth for the current version, currently `3.0.0`. It is copied verbatim into `dist/VERSION` during the build. See [conventions.md](conventions.md) for the rules on when to bump it.

## Testing and linting

There is no automated test suite or linter configured in this repository. Validation is manual: check changes visually in a browser against `docs/examples/`, at desktop and mobile widths. See [conventions.md](conventions.md) for the full validation checklist.

## Image resources

Aptus-owned visual resources (such as form icons) are referenced by a path relative to the CSS file itself, for example `src/notification.css` references `../img/aptus/form-error.svg`. This assumes the fixed deployment layout described in "Consumer integration" above, where every consuming project's build copies Aptus CSS and images so they end up as siblings under the same `res/` root:

```text
res/
  css/   (dist/*.min.css)
  img/
    aptus/
  js/
```

Because the path is relative to the CSS file rather than absolute, this resolves correctly regardless of which subdirectory of the web root the consuming app is installed under (`/res/...` or `/claude/res/...`), with no configuration needed on the consuming project's side. Do not switch this back to an absolute path (`/res/img/aptus/...`) or to a custom-property-based override — it would only be necessary if a consuming project could not guarantee the `res/css` + `res/img` sibling layout, which is not the case today. The CSS does not bundle the image itself; a missing icon must not prevent the related message or form text from being displayed.
